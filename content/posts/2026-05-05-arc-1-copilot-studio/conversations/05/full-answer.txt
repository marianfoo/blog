Detail	Value
ST22 Dump	DBSQL_SQL_ERROR — 2026-05-02 08:38:28 UTC

SQL Code	-9300 (Internal SQLDBC error), preceded by -10807 (Connection reset by peer on USR02)
Root Cause	The program executes SELECT * FROM (p_table) with p_table defaulting to
'ZARC1_DEMO_NO_SUCH_TBL' — a table that doesn't exist. The dynamic
FROM clause passes the non-existent table directly to the HANA database layer, which raises an unhandled SQL error. No
TRY/CATCH, no table validation, no DDIC existence check.
Impact	Work process crash → short dump. The DB connection was torn down (rc=104: Connection reset by peer), causing a cascading error when the kernel tried to read T100 for the error message text.
🛡️ Defensive Fix Applied (3 layers)
Layer	What was added	Why
1. Injection guard	CL_ABAP_DYN_PRG=>check_table_name_str()	Validates p_table is a legitimate table name, not an SQL-injection payload
2. DDIC existence check	DDIF_FIELDINFO_GET with sy-subrc check	Catches non-existent tables before hitting the database — outputs a clean error message and
RETURNs
3. Exception handler	TRY / CATCH cx_sy_dynamic_osql_error	Safety net around the dynamic SELECT — if anything still goes wrong at the DB layer, it's caught gracefully instead of dumping
